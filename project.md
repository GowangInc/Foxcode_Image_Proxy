# Project: School Image Generation Webserver

## Summary

Build a LAN-hosted image generation webserver for a school environment. Students sign in with a display name, submit prompts, wait in a serial queue, and view their own gallery. Teachers/admins can approve student access, moderate prompts, manage the queue, manage trusted admin browsers, and publish approved images to a public gallery.

## Current Assumptions

- The server runs on a machine reachable from the school network.
- Students use name-based identity only.
- API keys remain server-side and are configured through environment variables.
- Generated images are stored locally by the webserver.
- Admin access requires a password plus a trusted browser token once trusted browsers exist.

## Required Features

### Student Sign-In and Approval

- Prompt for student name before allowing generation.
- Store a student token in browser localStorage after approval.
- Allow an existing approved student to return by token.
- Allow the teacher to approve, deny, kick, and block students.
- Keep a blocklist that prevents re-requesting access until unblocked.

### Image Prompt Form

- Provide a text prompt box.
- Submit prompts to a server endpoint.
- Validate that prompts are not empty.
- Show progress while generation is running.
- Show understandable errors if generation fails.
- Allow the user to choose a provider preference: auto, Foxcode/OpenAI-compatible, or Gemini.

### Queue

- Maintain a single serial queue for generation.
- Show live queue status to students.
- Show the full queue to admins.
- Keep queue data updated live through SSE events.
- Support queue cancelation for queued jobs.
- Enforce a configurable delay between jobs for pacing tests.

### Student Gallery

- Show only the images generated by the signed-in student.
- Include prompt text, time, provider label, and moderation status.
- Refresh after generation completes.
- Show retry for failed images.

### Public Gallery

- Show only completed, approved images.
- Accessible without auth.
- Support search and student filtering.
- Show newest images first.

### Admin Gallery and Moderation

- Show all generated images from all students.
- Include student name, prompt, creation time, provider, and moderation state.
- Allow approve, reject, hide-public, delete, retry, and clear operations.
- Show the moderation queue separately.
- Allow editing blocked terms and regex patterns.

### Admin Browser Trust

- Enroll a first trusted admin browser with password.
- Require trusted browser token for later admin access.
- Allow generating one-time add-browser codes from a trusted browser.
- Allow clearing all trusted admin browsers.
- Persist trusted browsers on the server.

## Image API Integration

Use the foxcode image endpoint described by the installed `foxcode-image` skill.

Environment/config keys:

- `FOXCODE_API_KEY` preferred for the OpenAI-compatible Foxcode endpoint
- `OPENAI_API_KEY` fallback for Foxcode endpoint
- `GEMINI_API_KEY` for Gemini-compatible endpoint when configured
- `GOOGLE_GEMINI_BASE_URL` for Gemini base URL
- `GEMINI_IMAGE_MODEL` for Gemini model selection
- `GEMINI_API_KEY_AUTH_MECHANISM` for Gemini auth style

Current generation defaults:

- Model: `gpt-image-2`
- Size: `1536x1024`
- Quality: `high`
- Retries: configurable

The API key must stay server-side only.

## Suggested Architecture

- Web frontend for student, admin, and public pages.
- Server routes for approval, moderation, queue, and gallery data.
- Local image directory for generated PNGs.
- JSON metadata store for image records.
- JSON persistence for approved student tokens, blocklist, admin tokens, and moderation filters.

## Operational Notes

- Bind to `0.0.0.0` for LAN access.
- Keep generated files out of source control.
- Default Photos backup directory is `~/Photos/image2_webserver/` unless overridden.
- Public gallery and admin gallery should always render newest-first.
- Queue delay is configurable with `QUEUE_DELAY_MS` and defaults to 10 seconds for pacing tests.
- Use SSE for live queue updates.

## Remaining Gaps / Watchouts

- Pending login requests are still in-memory and are lost on restart.
- Admin browser trust uses tokens, not hardware IDs.
- Browser hardware IDs are not available in standard web browsers.
- Prompt filters are keyword/pattern based, not a complete safety system.
- Queue is still serial; parallel generation is not implemented.
